# Go 调试方法

Go 代码出问题时，最浪费时间的方式不是“不会查”，而是乱查。

真正高效的调试通常不是一上来改代码，而是按顺序做下面几件事：

1. 先稳定复现
2. 再缩小范围
3. 再验证假设
4. 最后才修改

这份文档不讲“某一个坑”，而是讲一套通用的 Go 调试方法。

## 1. 先把问题复现稳定下来

如果一个问题你不能稳定复现，就很容易落入：

- 猜原因
- 碰运气改
- 改完以为好了
- 下次又出现

优先做的事情：

- 写出最小复现步骤
- 记录输入、环境、配置、时间条件
- 判断是“必现”“偶现”还是“高并发才出现”

如果能先落成最小单测，后面的排查效率会高很多。

## 2. 先看报错原文，不要急着改

很多问题其实报错已经说得很清楚，只是大家习惯性跳过去。

重点关注：

- 错误文本
- stack trace
- 哪一层 first failure
- 哪些是因，哪些只是后续连锁反应

尤其是并发问题、空指针、数组越界、channel 死锁，运行时信息通常已经给了非常有价值的线索。

## 3. 先最小化输入，再最小化代码路径

如果问题只在复杂场景出现，先尝试缩小它：

- 更小输入
- 更少 goroutine
- 更少数据量
- 更短调用链
- 更简单外部依赖

你要找的不是“原场景 100% 完整保留”，而是“最小还能触发 bug 的条件”。

这一步做得越好，后面越不容易迷路。

## 4. 用单测固定问题，比手工点来点去更高效

一旦能复现，尽量把它写成测试：

```bash
go test -run TestName -v ./...
```

好处是：

- 能重复执行
- 改动后能快速验证
- 防止回归

如果是偶发 bug，至少也要先写一个最小可执行脚本，而不是每次靠手工触发。

## 5. 打日志时要打“决策点”，不是乱打

低效日志：

```go
fmt.Println("here")
fmt.Println("there")
```

高效日志应该回答这些问题：

- 输入是什么？
- 分支为什么这么走？
- 哪个 goroutine 在处理？
- 当前状态是什么？
- 某个值第一次变坏是在哪？

推荐打印：

- 关键参数
- 状态变化前后
- 错误链
- goroutine 标识或请求 ID

不要只打“到这里了”，要打“为什么会到这里”。

## 6. 常用 `go test` 调试命令

对应可运行示例见：[../debug/README.md](../debug/README.md) 的 “对应文档第 6 条”。

### 只跑一个测试

```bash
go test -run TestParseUser -v ./...
```

### 只跑某一组子测试

```bash
go test -run 'TestParser/invalid_id' -v ./...
```

### 禁用缓存，强制重跑

```bash
go test -count=1 ./...
```

### 并发问题先跑 race

```bash
go test -race ./...
```

### 反复跑同一测试，抓偶发问题

```bash
go test -run TestFoo -count=100 -v ./...
```

这几条是日常最常用的一组排查命令。

## 7. 并发问题，优先用 `-race`

对应可运行示例见：[../debug/README.md](../debug/README.md) 的 “对应文档第 7 条”。

只要怀疑有：

- 共享变量乱写
- 读写无锁
- goroutine 时序问题

第一反应就应该是：

```bash
go test -race ./...
```

`-race` 不是万能，但它能极大提高并发 bug 的发现效率。

注意：

- `-race` 会变慢
- 某些边缘时序 bug 不一定稳定触发
- 但它通常是并发排查的第一手工具

## 8. 用 Delve 做断点调试

对应可运行示例见：[../debug/README.md](../debug/README.md) 的 “对应文档第 8 条”。

Go 最常用的调试器是 `dlv`。

### 调试可执行程序

```bash
dlv debug .
```

### 调试测试

```bash
dlv test ./pkg/foo
```

### 常用操作

- `b main.main` 设置断点
- `b file.go:42` 按行断点
- `c` 继续执行
- `n` 下一步
- `s` 单步进入
- `p varName` 打印变量
- `bt` 查看调用栈

适合用 `dlv` 的场景：

- 单次流程复杂
- 想看某个值在哪一步变坏
- 想确认条件分支为什么进入

## 9. 死锁、卡死、阻塞，先看 goroutine 堆栈

对应可运行示例见：[../debug/README.md](../debug/README.md) 的 “对应文档第 9 条”。

如果程序表现为：

- 不退出
- 无响应
- CPU 不高但卡住

优先看 goroutine stack。

常见办法：

- 程序 panic 时看 runtime stack
- 服务进程发送 `QUIT` 信号打印堆栈
- 用 `pprof` 看 goroutine profile

例如类 Unix 环境常用：

```bash
kill -QUIT <pid>
```

拿到堆栈后，先看：

- 谁在等锁
- 谁在等 channel
- 谁在系统调用里卡住

## 10. 排查空指针和越界，不要只盯着崩溃点

空指针、切片越界、map nil 写入这类错误，真正的问题往往出在更早的上游。

比如：

- 参数本来就没初始化
- 某个分支提前返回导致对象没填好
- 某个字段在并发下被重置了

所以不要只盯着 crash 的那一行，要反推：

- 这个坏值从哪来？
- 谁负责初始化它？
- 哪个分支让它漏掉了？

## 11. 排查 I/O、数据库、外部请求问题时，要区分“本地逻辑错”和“外部依赖慢/错”

很多看起来像代码 bug 的问题，其实是：

- 网络抖动
- 数据库锁等待
- 外部接口超时
- 配置不一致

这时要先拆分边界：

- 进入本模块前的输入是什么
- 调外部依赖前参数是什么
- 外部返回了什么
- 错误是在本地构造出来的，还是从外部带回来的

不要把跨系统问题全部当成本地逻辑问题。

## 12. 性能问题和逻辑问题要分开查

对应可运行示例见：[../debug/README.md](../debug/README.md) 的 “对应文档第 12 条”。

很多人一看到“慢”，就开始凭感觉改算法、改锁、改 SQL。

正确顺序应该是：

1. 先确认慢在哪
2. 再确认为什么慢
3. 最后才优化

Go 里性能排查最常用的工具链是：

- benchmark
- `pprof`
- `trace`

如果是性能问题，尽量不要只靠日志和猜。

## 13. 先验证假设，再改代码

调试时很容易产生“我觉得就是这里的问题”。

正确方式应该是：

- 形成一个明确假设
- 加最小验证
- 看结果是否支持假设

例如：

- 我怀疑某个 channel 没关闭
- 那就去打印发送方退出时机和关闭时机
- 而不是直接大改 channel 结构

没有验证的修复，通常只是换一种猜法。

## 14. 常用命令清单

如果你想直接在本仓库里跟着操作，可以从这里跳到可运行示例：[../debug/README.md](../debug/README.md)。

### 日常调试

```bash
go test -run TestName -v ./...
go test -count=1 ./...
go test -race ./...
```

### 断点调试

```bash
dlv debug .
dlv test ./pkg/foo
```

### 基准和性能入口

```bash
go test -bench . ./...
go test -bench . -benchmem ./...
```

### 栈和 profile

```bash
go tool pprof cpu.out
go tool trace trace.out
```

## 15. 一份实战排查顺序

当你遇到一个 Go 问题时，可以按这个顺序来：

1. 先确认是否能稳定复现。
2. 读清楚报错和 stack trace。
3. 缩小输入和调用路径。
4. 写成最小测试或最小脚本。
5. 加针对性日志，观察坏值第一次出现的位置。
6. 并发问题优先跑 `-race`。
7. 需要单步时用 `dlv`。
8. 死锁/卡死时先看 goroutine stack。
9. 性能问题转到 `pprof/trace`，不要靠猜。
10. 验证根因后再改代码，并回归测试。

## 16. 一份简短的调试清单

- 问题是否稳定复现？
- 报错原文和 stack trace 是否读全了？
- 是否已经缩小到最小复现？
- 是否有对应的单测或可重复脚本？
- 日志是否打在关键决策点？
- 并发问题是否跑过 `-race`？
- 卡死问题是否看过 goroutine stack？
- 性能问题是否用过 `pprof/trace`？

如果能按这套顺序来，Go 调试效率会明显高很多。
