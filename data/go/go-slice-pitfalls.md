# Go 切片避坑指南

切片是 Go 里最常用、也最容易踩坑的数据结构之一。

很多 bug 不是因为语法不会，而是因为对切片的底层行为理解不够。尤其是下面几类场景最容易出错：

- 子切片和原切片共享底层数组。
- `append` 可能直接改写原数据。
- 回溯、DFS、缓存结果时忘记拷贝切片。
- `copy`、`len`、`cap` 的语义理解错误。
- 删除元素、函数传参、`nil` 与空切片的行为差异。

这份文档重点不是罗列 API，而是集中讲切片的高频避坑点。

## 1. 先搞清楚切片是什么

切片本身不是数组，它更像一个“描述符”，内部大致包含三部分：

- 指向底层数组的指针
- 当前长度 `len`
- 当前容量 `cap`

可以先记住一句话：

> 复制一个切片变量，复制的是“切片头”，不是底层数据本身。

这就是为什么两个切片变量看起来互相独立，实际上却可能改到同一块内存。

## 2. `make` 里的 `len` 和 `cap` 容易搞反

这是最基础、也最常见的坑。

错误示例：

```go
src := []int{1, 2, 3}
dst := make([]int, 0, len(src))

copy(dst, src)
fmt.Println(dst) // []
```

问题不在 `copy`，而在 `dst` 的长度是 `0`。`copy` 只会复制到目标切片的 `len(dst)` 个位置，容量 `cap` 再大也没用。

正确写法：

```go
src := []int{1, 2, 3}
dst := make([]int, len(src))
copy(dst, src)
fmt.Println(dst) // [1 2 3]
```

记忆方式：

- `len` 决定“现在能访问多少个元素”
- `cap` 决定“后面还能扩多少”

## 3. 子切片会共享底层数组

错误示例：

```go
a := []int{1, 2, 3, 4}
b := a[1:3]

b[0] = 99
fmt.Println(a) // [1 99 3 4]
fmt.Println(b) // [99 3]
```

很多人以为 `b := a[1:3]` 是“拷贝了一份新数据”，其实不是。`b` 只是引用了 `a` 的一段视图。

避坑建议：

- 如果你只想读，子切片没问题。
- 如果你后面会修改它，而且不希望影响原切片，就先拷贝。

推荐写法：

```go
b := append([]int(nil), a[1:3]...)
// 或者使用 Go 1.23 中的 slices.Clone(a[1:3])
```

## 4. `append` 可能悄悄污染原切片

这是最隐蔽的一类坑。

示例：

```go
a := []int{1, 2, 3, 4}
b := a[:2]

c := append(b, 99)

fmt.Println(a) // 可能是 [1 2 99 4]
fmt.Println(b) // [1 2]
fmt.Println(c) // [1 2 99]
```

为什么会这样：

- `b` 和 `a` 共享底层数组。
- `b` 的容量可能还够。
- `append(b, 99)` 不一定新分配内存，它可能直接把数据写回 `a` 的底层数组。

这类问题在递归、回溯、分支计算里非常常见。

### 特别危险的写法

```go
next := append(cur[:i], cur[i+1:]...)
```

这行代码在很多题解里都能看到，但它经常会直接改坏 `cur` 本身，尤其在递归里会污染兄弟分支。

更安全的写法：

```go
next := make([]int, 0, len(cur)-1)
next = append(next, cur[:i]...)
next = append(next, cur[i+1:]...)
```

如果只是想限制子切片后续 `append` 不要污染原数组，也可以使用三下标切片：

```go
b := a[:2:2]
```

这会把 `b` 的容量限制为 `2`，后续再 `append` 时更容易触发重新分配。

## 5. 回溯、DFS、保存结果时一定要快照

这是算法题里最高频的坑之一。

错误写法：

```go
ans = append(ans, path)
```

表面上看是把当前路径存起来了，实际上只是把当前切片头塞进结果里。后面 `path` 继续变化，之前保存的结果也会一起变。

正确写法：

```go
tmp := append([]int(nil), path...)
ans = append(ans, tmp)
```

或者：

```go
tmp := make([]int, len(path))
copy(tmp, path)
ans = append(ans, tmp)
```

记住一句话：

> 只要后面还会改这个切片，保存结果时就要拷贝。

## 6. `copy` 只看长度，不看容量

很多人知道 `copy(dst, src)`，但忽略了一个关键事实：

> `copy` 复制的元素个数是 `min(len(dst), len(src))`

所以这类写法没有效果：

```go
src := []int{1, 2, 3}
dst := make([]int, 0, 3)
copy(dst, src)

fmt.Println(dst) // []
```

因为 `len(dst) == 0`。

正确写法：

```go
dst := make([]int, len(src))
copy(dst, src)
```

## 7. 切片作为函数参数时，改元素和扩容不是一回事

切片传参很容易让人产生错觉。

示例：

```go
func change(s []int) {
	s[0] = 100
	s = append(s, 4)
}

func main() {
	s := []int{1, 2, 3}
	change(s)
	fmt.Println(s)
}
```

这里要分开看：

- `s[0] = 100` 很可能会影响调用方，因为它改的是共享底层数组。
- `s = append(s, 4)` 不一定能让调用方看到，因为这里改的是函数内部那个切片变量本身。

避坑原则：

- 需要修改元素内容，可以直接传 `[]T`。
- 需要让调用方看到“长度变化后的切片”，就返回新切片。

推荐写法：

```go
func addOne(s []int) []int {
	return append(s, 1)
}
```

## 8. 边遍历边删除，很容易漏元素

错误示例：

```go
for i, v := range s {
	if v%2 == 0 {
		s = append(s[:i], s[i+1:]...)
	}
}
```

问题在于：删除后元素左移，但 `range` 的索引推进方式并不会帮你重新对齐，结果就容易漏检查。

更稳妥的方式有两种：

### 写法一：倒序删

```go
for i := len(s) - 1; i >= 0; i-- {
	if s[i]%2 == 0 {
		s = append(s[:i], s[i+1:]...)
	}
}
```

### 写法二：双指针原地过滤

```go
n := 0
for _, v := range s {
	if v%2 != 0 {
		s[n] = v
		n++
	}
}
s = s[:n]
```

如果数据量大、删除频繁，第二种通常更稳。

## 9. 小切片可能拖住大数组，造成内存占用不降

示例：

```go
buf := make([]byte, 1024*1024)
small := buf[:10]
```

虽然 `small` 只有 10 个字节，但它仍然引用着那块 1MB 的底层数组。只要 `small` 还活着，这块大内存就不一定能被回收。

如果你只想保留一小段内容，应该主动拷贝：

```go
small = append([]byte(nil), small...)
```

这类问题在：

- 读取大文件后只截取头部
- 网络报文解析后只留一小段
- 日志、缓存、消息处理中保存子串或子切片

非常常见。

## 10. `nil` 切片和空切片不完全一样

两者都可以 `len == 0`，也都可以安全 `append`，但它们并不完全等价。

示例：

```go
var a []int      // nil slice
b := []int{}     // empty slice

fmt.Println(a == nil) // true
fmt.Println(b == nil) // false
```

常见差异点：

- `nil` 切片判断为 `nil`，空切片不会。
- JSON 编码时，`nil` 可能输出 `null`，空切片通常输出 `[]`。

如果接口协议要求固定返回空数组而不是 `null`，这里要格外注意。

## 11. 切片不能直接用 `==` 比较内容

除了和 `nil` 比较，切片不能直接使用 `==` 比较元素内容。

错误示例：

```go
if a == b { // 编译错误
}
```

正确做法：

- 用循环逐个比较
- 或者使用标准库 `slices.Equal`

示例：

```go
import "slices"

ok := slices.Equal(a, b)
```

## 12. 常见安全写法

### 拷贝一个切片

```go
dst := append([]int(nil), src...)
```

或者：

```go
dst := make([]int, len(src))
copy(dst, src)
```

### 回溯中保存当前路径

```go
ans = append(ans, append([]int(nil), path...))
```

### 删除下标 `i`

```go
s = append(s[:i], s[i+1:]...)
```

注意：

- 如果后面还要依赖原切片内容，这个删除操作可能共享底层数组。
- 在循环里连续删元素时，要特别小心索引移动问题。

### 限制容量，避免误改原数组

```go
sub := s[l:r:r]
```

这不是最常用的语法，但在需要控制 `append` 行为时非常有用。

## 13. 实战记忆法

如果你不想每次都重新推导，可以记住下面这几条：

1. 切片复制，默认只复制“视图”，不复制数据。
2. 只要后面还会修改，保存结果时就拷贝。
3. 看到子切片，就先问自己：它是不是和原数组共享内存。
4. 看到 `append`，就先问自己：它会不会改到原切片。
5. 看到 `copy`，就先问自己：目标切片的长度是不是够。
6. 看到循环删除，就先问自己：索引会不会错位。

## 14. 一份简短的避坑清单

写切片相关代码前，可以快速过一遍：

- 这个切片是不是和别的切片共享底层数组？
- 这里的 `append` 会不会污染原数据？
- 我保存到结果集里的，是不是一个独立副本？
- `copy` 的目标切片长度够不够？
- 这里需要的是 `nil` 切片还是空切片？
- 我是不是一边遍历一边删除？
- 这个小切片会不会拖住一大块底层内存？

如果这几个问题都想清楚，Go 切片的大多数坑基本就避开了。
