## 3. 高频基础题

### 3.1 为什么很多后端团队会选 Go

**回答思路：**

- Go 的语法和工具链比较统一，团队协作成本低。
- goroutine 和 channel 让 I/O 密集型服务更容易写出高并发模型。
- 二进制部署简单，编译和交付成本低。
- 标准库很强，`net/http`、`context`、`testing`、`pprof` 这些能力都比较成熟。
- 语言特性克制，长期维护的大型服务更容易形成一致代码风格。

**加分点：**

- Go 不是“特性最多”的语言，但它对工程效率很友好。
- 它比较适合 API 服务、网关、微服务、基础设施组件，而不是所有场景都通杀。

**常见追问：**

- Go 和 Java 各自更适合什么场景？
- Go 的不足是什么？

### 3.2 `new` 和 `make` 的区别

**回答思路：**

- `new(T)` 分配一个类型为 `T` 的零值对象，返回 `*T`。
- `make` 只用于 `slice`、`map`、`channel`，返回的是初始化后的可用对象本身，不是指针。

**示例：**

```go
p := new(int)          // *int
s := make([]int, 0, 8) // []int
m := make(map[string]int)
ch := make(chan int, 1)
```

**常见追问：**

- 为什么 `map` 用 `new(map[string]int)` 之后还不能直接写？

### 3.3 `slice` 的底层是什么

**回答思路：**

- `slice` 本质上是一个描述符，里面至少包含指向底层数组的指针、`len`、`cap`。
- 多个切片可能共享同一个底层数组，所以修改元素可能互相影响。
- `append` 触发扩容后，可能换新数组，也可能继续复用原数组。

**面试时不要只说“动态数组”。**

更完整一点应该说：

- `slice` 是对数组的一层轻量封装
- 它本身不是数组
- 共享底层数组是很多 bug 的根源

**搭配阅读：**

- [Go slice 避坑指南](./go-slice-pitfalls.md)

### 3.4 `map` 为什么不是并发安全的

**回答思路：**

- Go 的普通 `map` 没有内建并发读写同步。
- 并发写或并发读写可能触发运行时错误，也可能产生数据竞争。
- 如果有共享访问，要么外层加锁，要么做所有权收敛，要么考虑 `sync.Map` 的特定场景。

**常见追问：**

- `sync.Map` 适合什么场景？

可以答：

- 读多写少、key 集合逐渐稳定、并发访问模式特殊时可以考虑。
- 它不是普通 `map + Mutex` 的无脑替代。

### 3.5 `defer`、`return`、具名返回值的顺序

**回答思路：**

- `return` 会先准备返回值。
- 然后按后进先出的顺序执行 `defer`。
- 最后函数真正返回。

如果是具名返回值，`defer` 可以在返回前修改它。

**经典例子：**

```go
func f() (x int) {
	defer func() {
		x++
	}()
	return 1
}
```

返回值是 `2`。

**常见追问：**

- `defer` 参数什么时候求值？

答：

- 在 `defer` 语句注册时就求值，不是在真正执行时求值。

### 3.6 interface 的底层语义是什么

**回答思路：**

- interface 可以理解成“动态类型 + 动态值”的组合。
- 一个 interface 为 `nil`，要求这两部分都为空。
- 如果动态类型不为空、动态值为空，它就不是 `nil`。

**经典坑：**

```go
var p *MyError = nil
var err error = p
fmt.Println(err == nil) // false
```

因为这时：

- 动态类型是 `*MyError`
- 动态值是 `nil`

**搭配阅读：**

- [Go interface 避坑指南](./go-interface-pitfalls.md)

### 3.7 什么时候用值接收者，什么时候用指针接收者

**回答思路：**

- 需要修改接收者状态时，用指针接收者。
- 结构体较大时，通常也用指针接收者，避免复制。
- 如果一个类型的大多数方法都用指针接收者，最好整体保持一致。

**加分点：**

- 含有 `sync.Mutex` 这类字段的结构体通常更应该用指针传递，避免复制同步原语。

### 3.8 错误处理为什么是 Go 的关键能力

**回答思路：**

- Go 把错误作为显式返回值，而不是主异常流。
- 这样做的代价是代码更显式，收益是控制流更清晰。
- 工程里要学会包装错误、保留上下文、区分业务错误和系统错误。

**好的回答示例：**

- 下层报错向上用 `%w` 包装。
- 上层用 `errors.Is` / `errors.As` 判断根因。
- 对外层接口统一做错误翻译，不直接暴露内部错误细节。

**搭配阅读：**

- [Go error 避坑指南](./go-error-pitfalls.md)

### 3.9 `context` 的正确用法是什么

**回答思路：**

- `context` 主要解决超时、取消、请求级元数据传递。
- 它应该作为第一个参数传递，沿调用链往下传。
- 不应该存进结构体长期持有，也不应该拿 `context.Value` 当万能参数包。

**常见追问：**

- 为什么创建了 `WithTimeout` 后通常要 `defer cancel()`？

答：

- 为了及时释放定时器和上下文链路相关资源，不只是“主动取消”。

**搭配阅读：**

- [Go context 避坑指南](./go-context-pitfalls.md)

### 3.10 channel 为什么不只是“队列”

**回答思路：**

- channel 同时承担数据传递和同步信号的职责。
- 无缓冲 channel 更像同步交接点。
- 有缓冲 channel 可以削峰和解耦，但不会凭空消除背压。
- `close(channel)` 的核心语义是“不再发送新值”，一般由最后一个发送者关闭。

**常见追问：**

- 怎么判断一个 channel 是否关闭？

答：

- 接收时用两值赋值：`v, ok := <-ch`。
- 业务上更重要的是设计谁负责关闭，而不是四处探测是否已关闭。

### 3.11 goroutine 为什么轻，但不能乱开

**回答思路：**

- goroutine 相比线程轻很多，但不是免费资源。
- 它仍然带来栈内存、调度成本、排查复杂度、生命周期管理问题。
- 真正的关键不是“开多少”，而是“什么时候退出、谁负责通知退出、是否会泄漏”。

**常见追问：**

- goroutine 泄漏最常见的来源是什么？

答：

- 后台 goroutine 不监听 `ctx.Done()`
- 向没人接收的 channel 持续发送
- 消费者退出后生产者还在继续发

### 3.12 `Mutex` 和 `RWMutex` 怎么选

**回答思路：**

- 不是“`RWMutex` 更高级”，而是要看读写比、临界区长度和竞争模型。
- 读远多于写、读操作确实值得并发时，`RWMutex` 才更容易收益明显。
- 写操作不少、临界区很短、共享状态简单时，普通 `Mutex` 往往更直接。

**加分点：**

- `RWMutex` 不支持读锁升级成写锁。
- 一旦 writer pending，新的 reader 会被拦住，这是它抑制 writer 饥饿的关键。

**搭配阅读：**

- [Go sync 避坑指南](./go-sync-pitfalls.md)
- [Go 锁与读写锁的底层实现解析](./go-mutex-rwmutex-source.md)

## 4. 高频深挖题

### 4.1 Go 的 GMP 调度模型是什么

**回答模板：**

- `G` 是 goroutine。
- `M` 是真正执行代码的 OS 线程。
- `P` 是执行 Go 代码所需的调度上下文。
- runtime 会把大量 `G` 调度到少量 `M` 上运行，而 `P` 的数量通常受 `GOMAXPROCS` 控制。

**加分点：**

- goroutine 轻量的本质不是没有成本，而是调度和栈模型比线程轻得多。
- 一旦 goroutine 数量过多、阻塞很多、共享状态复杂，仍然会出现明显调度成本。

**搭配阅读：**

- [Go 并发模型与 Runtime 心智模型](./go-runtime-concurrency.md)

### 4.2 Go 的 `Mutex` 底层做了哪些优化

**回答模板：**

- 无竞争时走 CAS 快路径。
- 有竞争时进入慢路径，会结合自旋、等待队列、semaphore 休眠 / 唤醒。
- 当前版本还区分正常模式和饥饿模式。
- 正常模式追求吞吐，饥饿模式控制尾延迟，避免某些 waiter 一直抢不到锁。

**一旦面试官继续追问：**

- 你可以进一步说 `state` 里包含 `mutexLocked`、`mutexWoken`、`mutexStarving` 和 waiter 计数。
- 真正实现现在在 `internal/sync`。

### 4.3 Go 的 GC 大致怎么工作

**回答模板：**

- Go 现在的 GC 可以概括成并发标记清扫，核心是三色标记思路。
- 它会尽量把大部分标记工作并发化，减少长时间 Stop-The-World。
- 但 GC 不是“完全无停顿”，仍然有 STW 阶段，只是通常已经压缩得比较短。

**如果问 GC 为什么会影响性能：**

- 分配太多临时对象会增加 GC 压力。
- 堆上对象越多，扫描和回收成本通常越大。
- 所以降低无意义分配、减少逃逸、复用热点对象，都可能改善性能。

### 4.4 什么是逃逸分析

**回答模板：**

- 编译器会分析变量是否只能在当前栈帧内安全使用。
- 如果不能，就会让它逃逸到堆上。
- 逃逸本身不是 bug，但堆分配多了会带来 GC 成本。

**常见触发逃逸的场景：**

- 返回局部变量地址
- 闭包捕获变量
- 接口装箱带来的保守处理
- 编译器判断生命周期超过当前栈帧

**常见追问：**

- 怎么看逃逸？

答：

```bash
go build -gcflags="-m"
```

### 4.5 Go 的内存模型里最关键的点是什么

**回答模板：**

- 并发正确性不能靠“我觉得先执行到这里”来保证。
- 必须依赖明确的同步先行关系，也就是常说的 happens-before。
- channel 发送 / 接收、锁的加解锁、原子操作等，都会建立同步关系。
- 如果多个 goroutine 并发访问同一份数据，其中至少一个在写，又没有正确同步，就是数据竞争。

**加分点：**

- `go test -race ./...` 是排查这类问题的第一手工具。

### 4.6 `sync.Pool` 是什么，适合什么场景

**回答模板：**

- `sync.Pool` 是一个临时对象池，用来降低热点路径里的短生命周期对象分配压力。
- 它适合复用像 `bytes.Buffer` 这种可重置对象。
- 它不适合承载真正的业务缓存，因为池中的对象随时可能被 GC 清掉。

**面试里很重要的一句：**

- `sync.Pool` 的目标是降低分配与 GC 压力，不是做强语义缓存。

## 5. 工程化与排障题

### 5.1 `go.mod` 和 `go.sum` 各负责什么

**回答模板：**

- `go.mod` 表达模块边界、Go 版本、依赖声明和替换规则。
- `go.sum` 负责记录模块内容和 `go.mod` 的校验和，保证依赖完整性。
- 两者都应该提交到版本库。

**如果继续追问：**

- `go get` 负责日常增加 / 升级 / 删除依赖。
- `go mod tidy` 负责补齐缺失依赖、清理无用依赖，不等于“升级全部依赖”。

**搭配阅读：**

- [Go Modules 与依赖管理详解](./go-mod-dependency-management.md)

### 5.2 线上 Go 服务出了问题，你通常怎么查

**回答模板：**

我一般按这个顺序来：

1. 先稳定复现或尽量缩小触发条件
2. 先看错误原文和 first failure
3. 再结合日志、trace、metrics 判断是慢、错、阻塞还是资源耗尽
4. 如果怀疑并发问题，优先跑 `-race` 或看 goroutine 栈
5. 如果怀疑性能问题，优先 benchmark / pprof / trace，而不是凭感觉改

**面试里这类题重过程，不重某条命令背诵。**

**搭配阅读：**

- [Go 调试方法](./go-debugging-workflow.md)
- [Go 性能分析与 Profiling 方法](./go-performance-profiling.md)

### 5.3 说说你对 Go 工程化的理解

**回答模板：**

- 我会先明确模块边界和目录结构，比如入口放 `cmd/`，内部实现放 `internal/`。
- transport、业务编排、数据访问边界要清晰，避免 handler 变成巨石函数。
- 配置、日志、超时、错误翻译、测试、发布流程要尽量标准化。
- 目标不是目录“好看”，而是让人容易找到入口、容易改、容易测、线上容易定位问题。

## 6. go-zero 项目案例答法

这一节不只是讲 go-zero 是什么，而是讲“面试官问你怎么在项目里用 go-zero”时，怎么回答更像真实做过。

### 6.1 场景设定：用户下单系统

假设项目结构是：

- 对外提供下单 REST API
- 内部有用户服务、商品服务、库存服务
- 服务之间通过 gRPC 通信
- 对外层用 go-zero REST
- 内部 RPC 也用 go-zero 生成骨架

### 6.2 面试官问：你为什么在项目里选 go-zero

**回答模板：**

- 团队项目是多服务协作，go-zero 给了比较统一的 REST / RPC 工程组织方式。
- `goctl` 降低了新服务和新接口的接入成本。
- `ServiceContext` 能把 DB、Redis、RPC client、配置集中装配，依赖关系比较清晰。
- `handler`、`logic`、`svc` 这套结构对团队协作比较友好，新人上手也快。
- 对于超时、限流、熔断、日志这些横切能力，go-zero 也有比较成熟的工程化思路。

### 6.3 面试官问：你们的下单接口链路是怎么走的

**回答模板：**

可以答成一条链：

1. 外部请求先进入 go-zero REST 服务
2. `handler` 负责参数解析和调用 `logic`
3. `logic` 先做业务编排，比如校验用户、校验商品、调用库存 RPC、写订单
4. 各类依赖都从 `ServiceContext` 拿
5. 如果依赖内部服务，走 go-zero 的 RPC client 或 gRPC client
6. 最后统一做错误翻译和响应返回

**加分点：**

- `handler` 不做复杂业务，复杂业务放在 `logic` 或进一步拆到领域服务
- `context` 会从入口一路传到下游 DB / RPC

### 6.4 一个更像真实项目的 go-zero 代码骨架

```go
type ServiceContext struct {
	Config        config.Config
	OrderModel    model.OrderModel
	InventoryRPC  inventory.Inventory
	UserRPC       user.User
}

type CreateOrderLogic struct {
	logx.Logger
	ctx    context.Context
	svcCtx *svc.ServiceContext
}

func NewCreateOrderLogic(ctx context.Context, svcCtx *svc.ServiceContext) *CreateOrderLogic {
	return &CreateOrderLogic{
		Logger: logx.WithContext(ctx),
		ctx:    ctx,
		svcCtx: svcCtx,
	}
}

func (l *CreateOrderLogic) CreateOrder(req *types.CreateOrderReq) (*types.CreateOrderResp, error) {
	userResp, err := l.svcCtx.UserRPC.GetUser(l.ctx, &user.GetUserReq{Id: req.UserID})
	if err != nil {
		return nil, err
	}

	_, err = l.svcCtx.InventoryRPC.ReserveStock(l.ctx, &inventory.ReserveStockReq{
		ProductId: req.ProductID,
		Quantity:  req.Quantity,
	})
	if err != nil {
		return nil, err
	}

	orderID, err := l.svcCtx.OrderModel.Create(l.ctx, userResp.Id, req.ProductID, req.Quantity)
	if err != nil {
		return nil, err
	}

	return &types.CreateOrderResp{OrderID: orderID}, nil
}
```

### 6.5 面试官继续追问：go-zero 项目里最容易做坏的地方是什么

**建议回答：**

- 只用了脚手架，但业务全堆进 `logic`，最后 `logic` 变成新的大泥球
- `handler` 不够薄，transport 层和业务层混在一起
- `ServiceContext` 里依赖过多但没人管边界
- 默认能力一股脑全开，却不理解超时、熔断和调用链的真实代价

## 7. gRPC 项目案例答法

### 7.1 面试官问：你们为什么内部通信选 gRPC

**回答模板：**

- 内部服务数量较多，契约稳定、跨语言可能性存在，gRPC 的 Protobuf 契约和代码生成比较合适。
- 相比 JSON over HTTP，二进制协议更紧凑，接口定义更强约束。
- deadline、metadata、status code、interceptor 这些能力在工程上更统一。
- 如果有 streaming 需求，gRPC 比普通 REST 更自然。

### 7.2 订单系统里的 gRPC 示例

#### `.proto`

```proto
syntax = "proto3";

package order.v1;

option go_package = "example.com/project/pb/orderv1;orderv1";

service OrderService {
  rpc CreateOrder(CreateOrderRequest) returns (CreateOrderResponse);
  rpc WatchOrder(WatchOrderRequest) returns (stream WatchOrderResponse);
}

message CreateOrderRequest {
  int64 user_id = 1;
  int64 product_id = 2;
  int32 quantity = 3;
}

message CreateOrderResponse {
  int64 order_id = 1;
}

message WatchOrderRequest {
  int64 order_id = 1;
}

message WatchOrderResponse {
  int64 order_id = 1;
  string status = 2;
}
```

#### Server 端示意

```go
func (s *OrderServer) CreateOrder(ctx context.Context, req *pb.CreateOrderRequest) (*pb.CreateOrderResponse, error) {
	if req.Quantity <= 0 {
		return nil, status.Error(codes.InvalidArgument, "quantity must be positive")
	}

	orderID, err := s.orderSvc.Create(ctx, req.UserId, req.ProductId, int(req.Quantity))
	if err != nil {
		if errors.Is(err, service.ErrInsufficientStock) {
			return nil, status.Error(codes.FailedPrecondition, "insufficient stock")
		}
		return nil, status.Error(codes.Internal, "create order failed")
	}

	return &pb.CreateOrderResponse{OrderId: orderID}, nil
}
```

### 7.3 面试官问：gRPC 服务里你最关注哪些点

**建议回答：**

- `context` 和 deadline 是否一路向下传递
- metadata 里有没有 request id、trace id、token 等必要信息
- status code 是否区分业务错误和系统错误
- 拦截器里是否做好日志、trace、metrics、鉴权
- 大响应是否需要改成 stream，而不是一次性塞超大 message

### 7.4 面试官问：gRPC 遇到超时或 `Unavailable` 怎么查

**回答模板：**

我一般会先看：

1. 客户端 deadline 是否过短
2. 服务端处理是否真的慢
3. 下游依赖是否卡住
4. metadata / trace 是否贯通
5. 代理层是否支持 HTTP/2
6. 连接、发现、负载配置是否异常

如果要再补一句：

- `DeadlineExceeded` 和 `Unavailable` 在排查路径上不一样，一个更像调用超时，一个更像链路不可达或实例不可用。

## 8. GORM 项目案例答法

你消息里写的是 `grom`，这里按更常见的 `GORM` 理解。

### 8.1 面试官问：你为什么会在 Go 项目里用 GORM

**回答模板：**

- 对中后台业务来说，GORM 能比较快地完成模型映射、CRUD、事务和常见查询开发。
- 它的链式 API、事务封装、`WithContext`、`Preload`、`Scopes` 对业务开发效率很友好。
- 但我不会把它当“屏蔽 SQL 的黑盒”，复杂查询、性能敏感链路还是会关注生成 SQL、索引命中和执行计划。

### 8.2 一个真实感更强的 GORM 场景：下单扣库存

#### 数据模型

```go
type Product struct {
	ID        int64     `gorm:"primaryKey"`
	Name      string    `gorm:"size:128;not null"`
	Price     int64     `gorm:"not null"`
	Stock     int64     `gorm:"not null"`
	CreatedAt time.Time
	UpdatedAt time.Time
}

type Order struct {
	ID         int64       `gorm:"primaryKey"`
	UserID     int64       `gorm:"index;not null"`
	ProductID  int64       `gorm:"index;not null"`
	Quantity   int64       `gorm:"not null"`
	TotalPrice int64       `gorm:"not null"`
	Status     string      `gorm:"size:32;index;not null"`
	Items      []OrderItem `gorm:"foreignKey:OrderID"`
	CreatedAt  time.Time
	UpdatedAt  time.Time
}

type OrderItem struct {
	ID        int64 `gorm:"primaryKey"`
	OrderID   int64 `gorm:"index;not null"`
	ProductID int64 `gorm:"index;not null"`
	Quantity  int64 `gorm:"not null"`
	Price     int64 `gorm:"not null"`
}
```

#### 初始化

```go
db, err := gorm.Open(mysql.Open(dsn), &gorm.Config{})
if err != nil {
	return err
}
```

#### 带事务的下单逻辑

```go
func (r *OrderRepo) CreateOrder(ctx context.Context, userID, productID, quantity int64) (*Order, error) {
	var order Order

	err := r.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
		var product Product
		if err := tx.Clauses(clause.Locking{Strength: "UPDATE"}).
			First(&product, "id = ?", productID).Error; err != nil {
			return err
		}

		if product.Stock < quantity {
			return ErrInsufficientStock
		}

		if err := tx.Model(&product).
			Update("stock", gorm.Expr("stock - ?", quantity)).Error; err != nil {
			return err
		}

		order = Order{
			UserID:     userID,
			ProductID:  productID,
			Quantity:   quantity,
			TotalPrice: product.Price * quantity,
			Status:     "created",
		}
		if err := tx.Create(&order).Error; err != nil {
			return err
		}

		item := OrderItem{
			OrderID:   order.ID,
			ProductID: product.ID,
			Quantity:  quantity,
			Price:     product.Price,
		}
		if err := tx.Create(&item).Error; err != nil {
			return err
		}

		return nil
	})
	if err != nil {
		return nil, err
	}

	return &order, nil
}
```

### 8.3 这段 GORM 代码里有什么面试加分点

你可以主动解释：

- 用 `WithContext(ctx)` 把超时、取消、trace 传到数据库层
- 用事务保证“扣库存 + 写订单”原子性
- 用 `SELECT ... FOR UPDATE` 风格的行级锁避免并发扣库存错乱
- 用 `gorm.Expr("stock - ?", quantity)` 避免先读后写产生覆盖

### 8.4 GORM 常见追问

#### 追问一：为什么你还需要关心 SQL

**回答建议：**

- ORM 解决的是开发效率，不会自动替你解决索引、慢 SQL、N+1 查询、事务边界这些问题。
- 所以我会关注 GORM 生成的 SQL，必要时用日志、Explain 和慢查询平台验证。

#### 追问二：`Updates(struct)` 和 `Updates(map)` 有什么差异

**回答建议：**

- `Updates(struct)` 默认不会更新零值字段。
- 如果你要把某个字段更新成 `0`、`false`、空字符串，往往要用 `map` 或 `Select`。

例如：

```go
db.Model(&order).Updates(map[string]any{
	"status":      "paid",
	"retry_count": 0,
})
```

#### 追问三：怎么处理 N+1 查询

**回答建议：**

- 如果明确要查关联对象，可以用 `Preload`。
- 但不是所有地方都无脑 `Preload`，否则又会把查询做重。
- 复杂列表页通常要按真实字段和场景设计查询，必要时自己写更精确的 SQL。

#### 追问四：分页怎么做

**回答示例：**

```go
func Paginate(page, size int) func(*gorm.DB) *gorm.DB {
	return func(db *gorm.DB) *gorm.DB {
		if page <= 0 {
			page = 1
		}
		if size <= 0 || size > 100 {
			size = 20
		}
		return db.Offset((page-1)*size).Limit(size)
	}
}
```

然后：

```go
var orders []Order
err := db.Scopes(Paginate(page, size)).
	Where("user_id = ?", userID).
	Order("id desc").
	Find(&orders).Error
```

### 8.5 GORM 面试里的几个坑点

- 误以为 ORM 就不用懂 SQL 和索引
- 不理解事务边界
- 不知道零值更新问题
- 不知道 `Preload` 可能引入额外查询成本
- 不知道 `WithContext` 的意义

## 9. 综合场景题

### 9.1 面试官问：你做过一次 Go 服务性能优化吗

**回答模板：**

可以按“现象 -> 定位 -> 原因 -> 优化 -> 结果”来讲。

示例：

1. 现象：接口 RT 在高并发下明显抖动
2. 定位：先用 benchmark 和 `pprof` 看热点，再看 goroutine 和 mutex profile
3. 原因：发现热点在 JSON 编解码和一段共享 map 的锁竞争
4. 优化：减少临时对象分配，缩小锁粒度，把部分共享状态改成局部聚合后批量提交
5. 结果：P95 降低多少，allocs/op 降低多少

**重点不是背工具名，而是体现你有方法论。**

### 9.2 面试官问：你排查过 goroutine 泄漏吗

**回答模板：**

- 我先确认 goroutine 数量是否持续增长。
- 再结合 goroutine profile / stack 看是哪类调用堆积。
- 最终发现是某个请求级 goroutine 没监听 `ctx.Done()`，或者发送方在往没人消费的 channel 里持续发数据。
- 修复时会补上退出条件，并用测试或压测复现验证。

### 9.3 面试官问：如果让你设计一个内部微服务调用链，你会关注什么

**回答模板：**

- 契约定义是否清晰，比如 `.proto` 设计和版本演进
- deadline 是否统一传递
- 错误码是否有清晰分层
- trace / request id 是否贯通
- 重试、熔断、限流是否基于幂等性和流量模型设计
- 数据库事务边界和 RPC 边界是否清楚，避免分布式长事务乱扩散

## 10. 项目介绍怎么讲更像“真的做过”

很多人面试翻车，不是不会做，而是项目介绍像流水账。

更好的表达结构：

1. 先讲项目背景和业务目标
2. 再讲你负责的核心模块
3. 再讲关键技术决策
4. 最后讲结果、收益和你踩过的坑

### 10.1 一个更好的项目介绍模板

比如你做的是订单系统，可以这样说：

> 我负责的是订单中心和库存预占这一段链路。对外我们用 go-zero 暴露 REST API，对内服务间调用主要走 gRPC。数据层是 MySQL，业务开发里使用了 GORM 做主 CRUD 和事务封装。我的工作重点是把下单链路拆成清晰的 handler / logic / repository 分层，同时把超时、错误码、库存扣减一致性和链路排障这几块补稳。后面我们还针对锁竞争和对象分配做过一轮性能优化，把核心接口的 P95 延迟压下来了。

这类介绍的好处是：

- 有技术栈
- 有职责边界
- 有关键问题
- 有结果感

### 10.2 面试官继续追问：你在项目里最有价值的一次改动是什么

建议按这个结构答：

- 改之前的问题是什么
- 为什么这是核心瓶颈
- 你是怎么定位的
- 你改了哪些关键点
- 结果怎么验证
